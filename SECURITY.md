# Khora Security Policy

## Scope

This policy covers the Khora source tree, local artifacts generated by the framework, and the project documentation.

## Authorized Use Only

Khora must only be used on:

- systems you own and control
- systems where you have explicit written authorization
- isolated lab or training environments

The project should not be used for unauthorized access, disruption, persistence, or data extraction.

## Project Security Principles

- Validate user input before use
- Prefer least privilege
- Keep logs local
- Avoid embedding credentials in code or docs
- Keep documentation aligned with actual implementation

## Responsible Disclosure

When reporting a security issue in the codebase, include:

1. affected file or component
2. reproduction steps
3. impact assessment
4. suggested mitigation if known

## Local Data Handling

- Logs are stored locally in `logs/`
- Session artifacts are stored locally in `results/`
- Users remain responsible for protecting local output and cleaning up sensitive data

## Current Documentation Rule

Project documentation should describe the current codebase truthfully. Planned features must be marked as planned until they are implemented and testable.
